# Web3 工程系统的本质

> 来源：AI × Web3 School 开营直播回放
> 记录日期：2026-05-18

## 核心命题

> "Web3 工程系统的本质是：用密码学保证的、经济激励驱动的、去中心化信任的状态机。"

理解这个定义，比学会写合约更重要。

---

## 一、状态机：最核心的抽象

### Web3 = 全球共享的确定性状态机

```
当前状态（State）
    ↓ + 交易（Transaction）
新状态（New State）
```

**关键性质：**
- **确定性**：给定同一个初始状态和交易序列，任何人执行都得到相同结果
- **全局共享**：所有人看到的是同一个状态
- **不可回滚**：状态只能向前推进，不能回退

**类比（AI 视角）：**
- LLM 的推理是非确定性的（temperature > 0）
- Web3 执行是确定性的（EVM 无随机性，除非引入 Oracle）
- 这就是为什么 Web3 不能直接用 LLM 做链上决策

### 以太坊的状态包含什么

```
World State（世界状态）
├── 每个地址的账户状态
│   ├── Balance（ETH 余额）
│   ├── Nonce（交易计数，防重放攻击）
│   ├── Code（合约字节码，EOA 为空）
│   └── Storage（合约存储，键值对）
└── 每个区块的元数据
```

## 二、去中心化信任：替代了什么

### 传统系统的信任模型

```
用户 → 信任 → 公司 → 公司操作数据库
```

信任来源：品牌声誉、法律约束、监管合规

### Web3 的信任模型

```
用户 → 验证 → 代码 → 代码操作链上状态
```

信任来源：密码学 + 共识机制 + 公开可验证的代码

**本质差异：**
- 传统：信任人或机构，他们可以说谎、可以跑路、可以被黑
- Web3：信任数学和共识，代码公开、执行可验证、规则不可单方面修改

### 去信任化的代价

| 去掉的信任 | 付出的代价 |
|-----------|-----------|
| 不需要信任运营者 | 代码 bug 无法人工介入修复 |
| 不需要信任中间人 | 用户自己负责私钥安全 |
| 不需要许可 | 攻击者也不需要许可 |
| 规则透明 | 攻击面也透明 |

## 三、经济激励：系统运转的驱动力

### Gas 不只是手续费

Gas 是 **EVM 计算资源的定价机制**：

```
每个 EVM 操作码（opcode）有固定 Gas 消耗：
- ADD（加法）：3 Gas
- SSTORE（写存储）：20,000 Gas（最贵的之一）
- SLOAD（读存储）：2,100 Gas

总费用 = Gas Used × Gas Price（GWEI）
```

**设计意图：**
- 防止无限循环（循环每步消耗 Gas，耗尽则 revert）
- 为资源消耗定价（存储比计算更贵，因为永久占用节点存储）
- 防止垃圾交易（每笔交易都有最低成本）

### 矿工/验证者激励

```
验证者质押 ETH → 参与共识 → 出块获得 ETH 奖励
                             ↓
                         作恶被惩罚（Slash：罚没质押的 ETH）
```

这是系统不需要"信任验证者诚实"的原因：**不诚实的代价大于收益**。

### 应用层的经济激励设计

好的 Web3 系统设计，必须考虑：
- 用户参与的激励（为什么用我的协议）
- 攻击的代价（攻击是否无利可图）
- 流动性提供者的收益（DEX、借贷协议）
- 治理参与的激励（代币投票）

**AI × Web3 的新场景：** Agent 作为协议参与者，其行为也受经济激励约束（付 Gas、付服务费）。

## 四、密码学保证：系统的地基

### 三个核心密码学原语

```
哈希（Hash）：
→ 保证数据完整性（区块链式结构）
→ 地址生成
→ Merkle Tree 证明

非对称加密（Asymmetric Crypto）：
→ 私钥证明身份
→ 签名证明授权
→ 无需暴露私钥即可验证

零知识证明（ZK Proof，进阶）：
→ 证明"我知道某件事"但不透露"是什么"
→ ZK-Rollup 的核心机制
→ 隐私保护的技术基础
```

### 密码学如何让 Web3 工程成为可能

| 功能需求 | 密码学解决方案 |
|---------|--------------|
| 身份验证 | 私钥签名 |
| 防止篡改 | 哈希链 |
| 轻量级验证 | Merkle Proof |
| 隐私 + 可验证 | ZK Proof |
| 跨链资产证明 | 密码学承诺 |

## 五、工程系统的本质决定了什么

### 5.1 为什么"普通后端逻辑"不适用

```
传统后端思维 → Web3 实际情况
──────────────────────────────────────────
"出错了重试" → 重试可能造成双花或状态不一致
"加个 if 判断" → if 判断需要 Gas，每行代码都有成本
"记个日志" → 日志需要用 Events，不主动 emit 就没有
"加个配置开关" → 配置是链上状态，修改需要交易和 Gas
"先部署再优化" → 部署后不能修改，优化必须在部署前完成
```

### 5.2 AI 系统与 Web3 系统的根本差异

| 维度 | AI 系统 | Web3 系统 |
|------|---------|----------|
| 执行 | 概率性 | 确定性 |
| 错误 | 可重试，可忽略 | 失败是最终的 |
| 状态 | 可随时修改 | 只能追加 |
| 信任 | 信任模型提供者 | 信任数学 |
| 成本 | 按计算量/API 计费 | 按链上存储和计算计费 |
| 速度 | 毫秒级 | 秒到分钟级 |

### 5.3 AI × Web3 系统的正确接缝

AI 和 Web3 的协作应该在边界处理上特别小心：

```
AI 的输出（概率性）→ [人工确认] → 链上输入（确定性）

AI 适合做：
- 解释链上数据（理解交易含义）
- 生成链上操作的计划（用户决定是否执行）
- 分析合约代码（不能保证正确，但可以辅助）

AI 不适合做：
- 直接触发链上状态变更
- 保证操作结果（链上有很多不确定因素）
- 替代用户确认签名
```

## 个人总结

理解 Web3 工程系统的本质，改变了我对这个领域的认知：

Web3 不是"有区块链的后端"，而是一个**完全不同的系统范式**：
- 信任来自数学而非机构
- 状态是全局共享的而非私有的
- 执行成本是透明的而非隐藏的
- 错误是永久的而非可回滚的

学习 Web3 的核心挑战，不是学新的编程语言（Solidity 很简单），而是**建立一套完全不同的工程直觉**。
