# Web3 对安全要求很残酷

> 来源：AI × Web3 School 开营直播回放
> 记录日期：2026-05-18

## 核心命题

> "Web3 的安全要求是残酷的——没有热修复，没有回滚，没有客服，代码即法律。"

传统软件的容错机制在 Web3 几乎全部失效。

---

## 一、为什么"残酷"是准确的描述

### 传统软件 vs Web3 的安全容错对比

| 容错机制 | 传统软件 | Web3 |
|---------|---------|------|
| 发现 bug 后热修复 | ✓ 随时可以 | ✗ 合约不可修改（无代理模式时）|
| 回滚数据库 | ✓ 可以 | ✗ 链上状态永久 |
| 暂停服务 | ✓ 可以 | ✗ 合约无法停止（无 pause 功能时）|
| 客服处理 | ✓ 人工介入 | ✗ 无运营者，代码即法律 |
| 找责任方赔偿 | ✓ 可诉讼 | ✗ 匿名性 + 无法律管辖 |
| 内部测试环境 | ✓ 私有 | ✗ 测试网是公开的 |
| 错误静默处理 | ✓ try/catch 不影响用户 | ✗ revert 消耗 Gas，失败公开可查 |

**结论：Web3 的每一个 bug 都是公开的、可能是永久的、可能造成不可逆损失的。**

## 三、安全残酷性的具体表现

### 3.1 攻击速度

链上状态是公开的，机器人 24 小时监控：
```
漏洞发现 → 构造攻击交易 → 广播 → 执行
这个过程可以在同一个区块内完成（约 12 秒）
```

传统软件：漏洞被发现到被利用可能有几天到几周的时间窗口。
Web3：可能是零时间窗口。

### 3.2 攻击面是公开的

所有合约代码（已验证的）在 Etherscan 可见，所有交易在区块浏览器可查。
攻击者和防御者看到的信息是对称的。

### 3.3 用户无感知的风险

用户点击"Approve"时可能不知道：
- 授权了无限额度的 ERC-20 转账权
- 签名了允许第三方操作资产的消息
- 与未经审计的合约交互

```
常见攻击路径：
钓鱼网站 → 用户签名恶意消息 → 资产被转走
仿冒 DApp → 用户 Approve 恶意合约 → 资产被清空
```

### 3.4 AI 生成代码的额外风险

AI 会生成看起来正确但有安全漏洞的代码：

```solidity
// AI 可能生成这样的"正常"代码（有重入漏洞）
function withdraw() public {
    uint256 amount = balances[msg.sender];
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");
    balances[msg.sender] = 0;  // 状态更新在转账之后！
}
```

**AI 不知道这是错的，因为语法完全正确，逻辑看起来合理。**

## 四、安全防御体系

### 4.1 开发阶段

```
编写：遵循 Checks-Effects-Interactions 模式
使用：OpenZeppelin 经审计的标准合约
检查：Slither 静态分析（可在 Termux 安装）
测试：Foundry 模糊测试（Fuzzing）
审计：上线前专业审计（价格高但必要）
```

### 4.2 合约设计层面

| 机制 | 作用 |
|------|------|
| Ownable / AccessControl | 限制谁能调用敏感函数 |
| Pausable | 紧急暂停（有中心化取舍）|
| 升级代理（Proxy）| 允许逻辑升级（有风险，需谨慎）|
| 时间锁（Timelock）| 重要操作延迟执行，给社区反应时间 |
| 多签（Multisig）| N-of-M 签名才能执行高权限操作 |
| 限额（Rate Limit）| 单次操作或单日最大转账额度 |

### 4.3 AI × Web3 系统的额外安全要求

```
1. AI 不能持有私钥
2. AI 生成的合约代码必须人工 review + 工具检查
3. 涉及资产操作的 AI 建议必须人工确认
4. AI 不能解释"为什么可以信任这个合约"（它可能编造理由）
5. Tracing：记录 AI 做了什么决策，链上发生了什么
```

## 五、安全思维的本质转变

> "传统开发：先做功能，再考虑安全。Web3 开发：安全是第一设计约束，功能在安全边界内实现。"

**对新手的实践建议：**

1. 所有学习实验在测试网进行，绝不在主网测试
2. 任何 approve 操作前，检查授权额度（不要无限 approve）
3. 遇到不理解的合约交互，先查 Etherscan 上的代码
4. AI 生成的合约，先运行 Slither，再部署测试网，最后才考虑主网

## 个人理解

"残酷"不是在吓人，是在帮助建立正确的心理模型：
- 测试网随便玩，主网每次操作都当作不可逆的
- 对不理解的操作说"不"，不是谨慎，是正确
- AI 生成的代码是起点，不是终点
