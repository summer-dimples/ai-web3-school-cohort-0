# Vibe Coding AI 辅助编程

> 参考：https://aiweb3.school/zh/handbook/ai/vibe-coding/
> 学习日期：2026-05-19

## 什么是 Vibe Coding

"Vibe Coding"是 Andrej Karpathy（前 Tesla AI 负责人，OpenAI 联合创始人）提出的概念：

> 用自然语言描述意图，让 AI 写大部分代码，人负责方向、审查和验证。

这不是让 AI 替代开发者，而是改变开发者的工作重心：**从写代码 → 描述意图 + 审查代码**。

## 主流工具

| 工具 | 特点 | 适合 |
|------|------|------|
| **Claude Code** | 深度代码理解，Terminal 内操作，可读写文件 | 复杂项目，文件级操作 |
| **Cursor** | IDE 级集成，多文件上下文，Tab 补全 | 日常开发，大型项目 |
| **GitHub Copilot** | 行内补全，IDE 插件 | 代码补全，文档 |
| **Aider** | 终端工具，Git 集成 | 命令行工作流 |
| **Windsurf** | Codeium 出品，全文件上下文 | 类 Cursor 替代 |

本课程主力工具：**Claude Code**（已在第一天配置）。

## 人的角色不是消失，而是转变

| 之前 | 之后 |
|------|------|
| 写样板代码 | → AI 生成，人审查 |
| 查 API 文档 | → AI 查询并给出示例 |
| 调试语法错误 | → AI 定位，人确认方案 |
| **架构决策** | → **人决定**（AI 只能建议）|
| **安全审查** | → **人负责**（AI 会漏洞）|
| **测试设计** | → **人主导**（测什么比怎么测更重要）|
| **业务逻辑验证** | → **人确认**（AI 不懂业务语境）|

## AI 输出的 5 类风险

用 AI 写代码时，必须理解哪些输出不可靠：

| 风险类型 | 描述 | 应对 |
|---------|------|------|
| **事实错误** | 自信地编造不存在的 API、函数签名 | 运行测试验证，查官方文档 |
| **引用错误** | 编造论文、链接、数据来源 | 不信任 AI 给的链接，自行搜索 |
| **推理漂移** | 长上下文中逻辑链断裂，结论偏离前提 | 分段验证，拆小任务 |
| **执行越权** | Agent 做了超出授权范围的操作 | 设置 guardrails，保留人工确认 |
| **工具误用** | 调用错误工具或参数 | 用 tracing 监控，review 工具调用 |

**Web3 场景下格外重要：** AI 生成的合约代码必须经过安全审查，不能直接部署到主网。

## 高效使用 Claude Code 的实践

### 给出充分上下文

```
❌ 差：帮我写一个合约
✓ 好：帮我用 Solidity 0.8.20 写一个 SimpleStorage 合约，
       包含 set(uint256) 和 get() 函数，
       部署在 Sepolia 测试网，需要符合 OpenZeppelin 安全标准
```

### 增量式开发，不要一次让 AI 写完

```
步骤 1：先让 AI 写合约结构
步骤 2：确认无误后，让 AI 写测试
步骤 3：确认测试覆盖后，让 AI 写部署脚本
步骤 4：人工 review 全部代码后部署
```

### 让 AI 解释它写了什么

```
解释这段代码中 unchecked 块的作用，以及什么情况下会有溢出风险
```

### 明确要求 AI 指出不确定的地方

```
如果你不确定某个实现，请明确告诉我，不要猜测
```

## Vibe Coding 的边界

**绝对不做（AI 不适合的）：**
- 跳过代码 review 直接部署
- 让 AI 生成私钥相关逻辑后直接使用
- 在生产环境中跳过测试
- 相信 AI 给的安全保证

**可以完全让 AI 做：**
- 样板代码、重复性结构
- 解释陌生代码库
- 文档生成
- 单元测试骨架
