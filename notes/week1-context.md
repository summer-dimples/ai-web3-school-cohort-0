# Context 上下文管理

> 参考：https://aiweb3.school/zh/handbook/ai/context/
> 学习日期：2026-05-19

## 上下文窗口是什么

上下文窗口（Context Window）是模型在一次推理时**能看到的全部文本**，相当于模型的"工作内存"。超出窗口的内容模型完全看不到。

```
┌────────────────────────────────────────────┐
│  上下文窗口（例：200k tokens）               │
│  ┌──────────┐ ┌────────┐ ┌──────────────┐  │
│  │System    │ │历史对话 │ │ 工具输出/检索 │  │
│  │Prompt    │ │        │ │ 文档/当前任务│  │
│  └──────────┘ └────────┘ └──────────────┘  │
└────────────────────────────────────────────┘
```

## 主流模型的上下文大小（2025）

| 模型 | 上下文窗口 |
|------|----------|
| GPT-4o | 128k tokens |
| Claude Sonnet 4.6 | 200k tokens |
| Gemini 1.5 Pro | 1M tokens |
| GLM-4 | 128k tokens |

**200k tokens ≈ 约 15 万汉字 ≈ 一本中型小说**，听起来很大，但一个大型代码仓库 + 长对话 + 工具输出很快就填满了。

## 上下文里放什么

| 位置 | 内容 | 管理策略 |
|------|------|---------|
| 开头（System Prompt）| 身份、规则、工具定义 | 保持精简，每次都占用 |
| 中间（历史对话）| 之前的问答 | 滑动窗口 / 摘要压缩 |
| 注入内容 | RAG 检索文档、工具结果 | 按需注入，用后丢弃 |
| 末尾（当前任务）| 用户最新请求 | 最重要，模型注意力最强 |

## "Lost in the Middle" 问题

研究发现：把关键信息放在上下文**中间**，模型更容易忽略；放在**开头或结尾**效果最好。

**实践建议：**
- 关键指令放在系统提示开头 或 用户消息结尾
- 长文档 RAG 时，检索到的最相关段落放在最后
- 不要在上下文中间塞太多不相关内容

## 上下文管理策略

### 策略 1：摘要压缩
当对话过长时，用模型总结之前的对话，用摘要替换原始历史：

```
原始：10轮对话 = 5000 tokens
压缩后：摘要 = 300 tokens + 最新2轮 = 1000 tokens
节省：3700 tokens
```

### 策略 2：滑动窗口
只保留最近 N 轮对话，丢弃更早的历史。简单但会丢失早期上下文。

### 策略 3：RAG（检索增强）
不把全部知识放进上下文，而是按需检索相关片段注入。见 `week1-rag.md`。

### 策略 4：结构化状态
用结构化数据（JSON/YAML）存储关键状态，而非保留完整对话历史：

```json
{
  "wallet_address": "0x1234...",
  "network": "sepolia",
  "task": "deploy SimpleStorage",
  "step": 3
}
```

## 四个控制层面

从课程大纲总结的核心框架：

| 层面 | 控制什么 |
|------|---------|
| **上下文窗口** | 模型能看到多少信息（信息边界）|
| **系统指令** | 模型的身份、语气和行为边界 |
| **提示词** | 当前任务的具体意图 |
| **工具调用** | 让模型从"说话"变为"做事" |

理解这四层是设计可靠 AI 系统的基础。
