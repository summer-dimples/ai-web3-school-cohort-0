# RAG 检索增强生成

> 参考：https://aiweb3.school/zh/handbook/ai/rag/
> 学习日期：2026-05-19

## RAG 解决什么问题

LLM 有两个根本限制：
1. **知识截止日期**：训练数据有时效性，不知道最新事件
2. **幻觉**：对训练不足的专有知识会"编造"

RAG（Retrieval-Augmented Generation）的核心思路：**不把知识存在模型里，而是存在外部数据库里，需要时检索出来注入上下文。**

```
用户提问
    ↓
把问题转成向量（Embedding）
    ↓
在向量数据库中搜索最相关的文档片段
    ↓
把检索到的片段注入上下文
    ↓
LLM 基于这些真实文档回答
    ↓
输出（有来源依据，可溯源）
```

## 核心组件

### 1. 文档切分（Chunking）

把长文档切成小块，每块独立嵌入：

| 策略 | 说明 | 适合场景 |
|------|------|---------|
| 固定大小 | 每 512 tokens 一块 | 简单，通用 |
| 语义切分 | 按段落/章节边界 | 文档结构清晰时 |
| 递归切分 | 先大块再细分 | 层次文档 |
| 重叠切分 | 相邻块有 50-100 token 重叠 | 避免截断关键信息 |

**重叠是关键**：防止一个句子被切成两块而丢失语义。

### 2. 向量嵌入（Embedding）

把文本转成数字向量，语义相近的文本在向量空间中距离近：

```python
from anthropic import Anthropic
# 或使用 OpenAI / BGE / text2vec 等嵌入模型
embedding = embed_model.encode("什么是 Gas？")
# → [0.23, -0.15, 0.87, ...]  维度通常 768 或 1536
```

### 3. 向量数据库

存储和检索向量的专用数据库：

| 数据库 | 特点 | 适合 |
|--------|------|------|
| Chroma | 轻量，本地 | 原型/小项目 |
| Pinecone | 托管，易用 | 生产环境 |
| Weaviate | 混合搜索 | 复杂查询 |
| FAISS | Meta 开源，极快 | 大规模本地 |
| pgvector | PostgreSQL 扩展 | 已有 PG 的项目 |

### 4. 检索策略

- **语义搜索**：向量相似度（cosine / dot product）
- **关键词搜索**：BM25，精确匹配
- **混合搜索**：两者结合，效果更好
- **重排序（Reranking）**：用交叉编码器对初步检索结果重新排序

## RAG vs 微调：选哪个？

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 实时（更新数据库即可）| 需要重新训练 |
| 可溯源 | 是（能给出来源）| 否 |
| 成本 | 推理时有检索开销 | 训练成本高 |
| 适合 | 知识库问答、文档搜索 | 风格/格式/特定能力适配 |

**经验法则：** 知识是动态变化的 → 用 RAG；需要改变模型行为/风格 → 用微调。

## Web3 场景下的 RAG 应用

- 合约文档问答：把 ABI、文档、审计报告做成知识库
- 交易解释：检索相关协议文档解释链上数据
- 安全检查：检索已知漏洞库，对新合约进行比对

## RAG 失效的场景

- 问题和文档的表达方式差异很大（语义不匹配）
- 文档质量差或更新不及时
- 需要跨多个文档综合推理（单块检索不足）
- 知识本质上就没有在数据库里
