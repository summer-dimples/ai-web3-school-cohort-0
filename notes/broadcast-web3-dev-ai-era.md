# AI 时代 Web3 开发者的基础知识与架构能力

> 来源：AI × Web3 School 开营直播回放
> 记录日期：2026-05-18

## 核心命题

> "AI 时代的 Web3 开发者，不只是会写合约，而是能驾驭 AI 工具、理解系统全局、在不确定性中做出正确架构决策的人。"

这句话的逆命题同样成立：**只会用 AI 工具但不懂 Web3 本质的人，会做出危险的系统。**

---

## 一、Web3 开发者需要哪些基础知识

### 1.1 不可缺少的底层认知

| 知识域 | 为什么必须理解 | 只靠 AI 的风险 |
|--------|--------------|--------------|
| 密码学原理 | 签名、地址、私钥的安全边界 | AI 可能生成密钥管理漏洞 |
| 链上状态模型 | 合约执行和存储的代价 | AI 不理解 Gas 经济，会写低效合约 |
| 共识机制 | 最终确定性、重组风险 | 错误假设交易"立刻完成" |
| 经济激励设计 | 协议可持续性和攻击面 | AI 无法理解博弈均衡 |
| 智能合约不变性 | 部署后无法修改 | AI 生成的代码 bug 无法热修复 |

**结论：Web3 的基础知识没有捷径。AI 可以帮你写代码，但不能替你建立安全直觉。**

### 1.2 需要新增的 AI 时代能力

| 能力 | 描述 |
|------|------|
| Prompt 工程 | 精确描述需求，让 AI 生成高质量代码和分析 |
| AI 输出验证 | 知道 AI 在哪里会出错，主动校验关键输出 |
| 系统编排 | 设计 AI + 链上工具的协作流程 |
| 人工确认节点设计 | 在自动化流程中精准插入人工审查点 |
| 可观测性 | Tracing AI 行为和链上行为，两个层面都要可查 |

## 二、架构能力：AI 时代更重要，不是更不重要

### 2.1 为什么架构能力反而升值了

AI 能替代的：写样板代码、查文档、生成测试用例。

AI 替代不了的：
- **为什么这样设计**（trade-off 判断）
- **系统如何失败**（failure mode 分析）
- **边界在哪里**（安全边界 + 权限边界）
- **如何演化**（升级策略、迁移方案）

> AI 工具让初级开发者能写出代码，但让架构能力的稀缺性更突出——因为现在"写代码"这件事的门槛降低了，而"设计正确系统"的门槛没有降低。

### 2.2 Web3 系统架构的独特挑战

```
传统系统架构                    Web3 系统架构
────────────────────────────────────────────────
可热修复                →      部署即定局，升级需代理模式
数据库私有              →      链上状态全公开
服务器可关闭            →      合约无法停止
错误可回滚              →      失败交易不可撤销（Gas 已消耗）
权限集中管理            →      权限在合约逻辑中，公开可验证
成本相对固定            →      Gas 随网络拥堵动态波动
```

### 2.3 AI × Web3 系统架构的三层模型

```
┌─────────────────────────────────────────────┐
│  Layer 3：应用层                              │
│  用户界面、Agent 交互、DApp 前端              │
├─────────────────────────────────────────────┤
│  Layer 2：AI 编排层                           │
│  Prompt、Workflow、Agent、工具调用、人工确认   │
├─────────────────────────────────────────────┤
│  Layer 1：链上执行层                          │
│  智能合约、签名、Gas、状态、事件              │
└─────────────────────────────────────────────┘

关键原则：每一层的安全边界不能依赖上层保证
→ Layer 1 合约不能信任 AI 的输入
→ Layer 2 Agent 不能假设链上操作成功
→ Layer 3 界面不能绕过 Layer 1 的权限检查
```

## 三、从"写代码"到"驾驭系统"

### AI 时代开发者的新工作流

```
旧工作流：需求 → 写代码 → 测试 → 部署
新工作流：需求 → 设计架构 → 指挥 AI 生成 → 审查验证 → 测试 → 部署
```

**变化的：** 大量重复性代码交给 AI
**不变的：** 架构决策、安全审查、测试覆盖策略、业务逻辑验证

### 在 Web3 场景下，"驾驭 AI"的额外要求

1. **不让 AI 接触私钥**（工具权限设计）
2. **不让 AI 决定是否签名**（人工确认强制节点）
3. **不信任 AI 生成的地址和哈希**（必须链上验证）
4. **AI 生成的合约必须安全审查**（Slither/人工 review）

## 个人思考

这个演讲的核心矛盾是：AI 工具让 Web3 开发门槛降低，同时让错误的代价更难被发现（生成速度快但验证容易被跳过）。

**我的位置：** AI 有基础，Web3 新手。优先建立 Web3 安全直觉，而不是急着用 AI 生成合约代码。
